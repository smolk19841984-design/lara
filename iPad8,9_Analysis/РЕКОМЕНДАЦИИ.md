## ✅ КРИТИЧЕСКИЕ ИСПРАВЛЕНИЯ ПРИМЕНЕНЫ (2026-04-10)

### 1. **OFFSET ФИКС** — Invalid TRO в remote call
**Проблема**: TRO values `0x2f00`, `0x5b00`, `0x0`, `0x1f00` (не kernel pointers)

**Анализ**: Созданы скрипты `offset_hypothesis_tester.py` и `deep_8ksec_analyzer.py`
для анализа паттернов и исследования 8kSec.io. Выявлено, что MIE не применим к A12X.

**Фикс применён**:
- `rc_off_task_threads_next = 0x48` (был 0x58)
- `rc_off_thread_t_tro = 0x348` (был 0x358)
- `rc_off_thread_task_threads_next = 0x348`

**Ожидаемый результат**: TRO должны стать валидными kernel pointers (0xfffffff0...)

### 2. `proc_task()` — неверный способ получения task из proc
**Файлы**: `jbdc/kexploit/rc_kutils.m`, `lara/kexploit/rc_kutils.m`

Старый код читал `proc_ro + 0x08` как `pr_task`. Darksword использует
исключительно `proc + PROC_STRUCT_SIZE`. Это подтверждено логом:
```
our_proc = 0xffffffdf07547d48
our_task = 0xffffffdf07548478
diff = 0x730 = PROC_STRUCT_SIZE  ← ТОЧНОЕ СОВПАДЕНИЕ
```
**Исправление**: `proc_task()` теперь использует `taskbyproc(proc)` как
первичный метод. `proc_ro->pr_task` — только для верификации с логированием.

### 3. `init_remote_call` — бесконечный цикл
**Файлы**: `jbdc/TaskRop/RemoteCall.m`, `lara/TaskRop/RemoteCall.m`

Цикл обхода тредов `while (chain && chain != sentinel && retryCount < 10)`
не имел защиты от бесконечного повторения при `validThreadCount == 0`.
**Исправление**:
- `int maxIter = 1024` + `--maxIter > 0` в условие `while`
- `rc_probe_tro_offset(dummyThreadAddr)` вызывается ПЕРЕД чтением `dummyThreadTro`
- Диагностический PRE-WALK лог для определения правильности `task_addr`
- Проверка `nextChain`: если не kernel ptr и не sentinel — `break`

### 4. `vfs_bypass_mac_label` — ncache не прогрет
**Файл**: `jbdc/kexploit/vfs.m`

`resolvepath()` не может найти vnode для свежесозданного файла — ncache пуст.
**Исправление**: `stat()` warm-up + retry через `usleep(5000)` перед `resolvepath()`.
Также добавлен scan ±16 байт от `OV_LABEL` если root_label-pointer невалиден.

#### 🚨 КРИТИЧЕСКАЯ ПРОБЛЕМА: Invalid TRO Values
**Симптомы после тестирования:**
- TRO values: `0x2f00`, `0x5b00`, `0x0`, `0x1f00` (не kernel pointers >= 0xffffff8000000000)

**Возможные причины:**

2. `rc_off_thread_t_tro = 0x358` неправильный (хотя найден пробой)

В этой папке подготовлена автоматизированная среда для анализа патчей ядра, как было описано в **8kSec CVE-2024-23265 Patch Diffing**.

## Окружение
- **Устройство:** `iPad8,9`
- **Прошивка 1 (Уязвимая):** `17.3.1 (21D61)`
- **Прошивка 2 (С патчем):** `17.4 (21E219)`
- **Целевой модуль (KEXT):** `com.apple.driver.AppleDiskImages2` (пример из статьи).

## Шаг 1: Автоматическая Загрузка и Извлечение Ядра
Скрипт `automate_ipsw.py` делает ровно то, что было описано в статье:
1. Загружает `kernelcache` для **iPad8,9** (21D61 и 21E219).
2. Выгружает уязвимый и пропатченный драйвер `AppleDiskImages2` в отдельные папки.
3. Снимает системные вызовы ядра.

**Чтобы запустить скрипт, выполните в терминале:**
```bash
python iPad8,9_Analysis\automate_ipsw.py
```
*(Загрузка `kernelcache` займет несколько минут в зависимости от скорости интернета).*

## Шаг 2: Символизация в Ghidra (Как в статье)
1. Выгруженные скриптом файлы `.kext` вам нужно загрузить в **Ghidra**. То есть создайте 2 проекта (например, `iOS_17_3_1_Kext` и `iOS_17_4_Kext`).
2. В статье упоминается утилита **symbolicator** ([blacktop/symbolicator](https://github.com/blacktop/symbolicator/)). С помощью команды `ipsw kernel sym kernelcache.release... --json` вы можете сгенерировать JSON со всеми символами ядра.
3. Этот JSON нужно загрузить через `Symbolicate.java` в Ghidra.

## Шаг 3: Diffing (Поиск смещений и исправлений)
Как было указано в 8kSec, после прохода `DecompInterface` или просто экспорта всего псевдокода (C) из функции, мы можем использовать **Meld** для нахождения того, куда Apple добавила проверки `(lVar4 != 0) && (lVar4 != -1)`.

Тот факт, что вы всё выделите в эту изолированную папку, защитит эксплойт-билд (`lara`) от поломок и позволит вам сгенерировать оффсеты для любых функций (типа `task_get_ipc_port_kobject` или `allproc`), пропуская это ядро через дизассемблер и парсер системных вызовов.
