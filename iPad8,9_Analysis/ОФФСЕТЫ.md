### Ключевые соотношения (верны для всех iOS 17.x / A10–A17):
```
task_threads_next  = t_tro         (0x348 ✓)
guard_exc_info     = t_tro - 0x50   (0x348 - 0x50 = 0x2f8)
ctid               = t_tro + 0xb0   (0x348 + 0xb0 = 0x3f8)
```

### ✅ ПРОБЛЕМА РЕШЕНА: Invalid TRO в remote call
**Предыдущие симптомы:**
- TRO values: `0x2f00`, `0x5b00`, `0x0`, `0x1f00` (не kernel pointers)
- first_thread: `0xffffffdf02fdb380`
- first_tro: `0x2f00` (invalid)

**Применённый фикс:**
1. `rc_off_task_threads_next = 0x48` (был 0x58)
2. `rc_off_thread_t_tro = 0x348` (был 0x358)
3. `rc_off_thread_task_threads_next = 0x348`

**Ожидаемый результат:**
- TRO values должны быть валидными kernel pointers (0xfffffff0...)
- Remote call должен успешно находить SpringBoard threads
- Jailbreak должен достичь 100% успеха

В процессе автоматизации мы создали инструмент `scripts/lara_offset_finder.py`, который динамически находит смещения (offsets) в любом ядре iOS (kernelcache) с помощью утилиты `ipsw`.

## Как это работает?
Вместо захардкоженных размеров структур (которые ломаются при обновлениях Apple) инструмент использует логику "**Эвристического поиска на базе дизассемблера**":

1. **Поиск базового адреса:** Инструмент извлекает `__TEXT base address` (например, `0xfffffff007004000`) из таблицы сегментов ядра.
2. **Извлечение таблиц:** Выгружаются таблицы системных вызовов (`syscalls`) и Mach-ловушек. Мы точно знаем адреса экспортируемых функций, таких как `setuid`.
3. **Эмуляция инструкций (ADRP+LDR):** Утилита дизассемблирует известные функции и анализирует паттерны загрузки страниц памяти (инструкция `ADRP`) и получения значений по смещению (инструкции `LDR` или `ADD`).

## Текущие результаты для iPad8,9 (iOS 17.3.1 - сборка 21D61)

### 1. `allproc`
Переменная `allproc` хранит список всех процессов в системе. Она необходима джейлбрейку для обхода процессов (чтобы найти PID и украсть/подменить task port). Мы знаем, что системный вызов `setuid` одним из первых обращается к структуре текущего процесса и списку процессов.

Инструмент нашел следующие потенциальные указатели в функции `setuid`:
* Кандидат 1: `0x3216cc8`
* Кандидат 2: `0x11c4cf0`

### 2. `rootvnode`
Указатель `rootvnode` нужен для работы с File System (VFS), маскировки точки монтирования `/` при рутинге и обхода песочницы (sandbox escape).
Его легко найти, проанализировав системный вызов `chroot()`, так как эта функция в XNU напрямую обращается к корневому vnode для его смены или валидации. 

Кандидаты для `rootvnode` в дизассемблере `chroot`:
* Кандидат 1: `0x31c3000` (скорее всего, это он, лежит в секции `__DATA`)
* Кандидат 2: `0x96b668`
* Кандидат 3: `0x32398b0`

### 3. `kernel_task`
Глобальная структура/задача `kernel_task` содержит порт самого ядра, через нее выполняются функции, доступные только Ring 0. 
В Mach-ловушке `task_for_pid` мы часто ищем сравнение запрашиваемого PID с ядром (или проверяем права). Указатель на задачу `kernel_task` загружается для проверок безопасности.
* Кандидат 1: `0x31ad400`
* Кандидат 2: `0x96b928`
* Кандидат 3: `0x968210`

*Обычно `allproc` и `rootvnode` лежат в секции `__DATA_CONST` или `__DATA`, и мы можем программно верифицировать правильный offset во время выполнения джейлбрейка, просто пройдясь по 2-3 кандидатам, найденным скриптом.*
