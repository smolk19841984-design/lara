### Корневые причины сбоев

#### 1. `init_remote_call` — КРИТИЧЕСКИЙ (ИСПРАВЛЕН)
**Симптом**: Цикл обхода тредов Spring Board выдаёт 100+ строк  
`SKIP invalid tro: 0x2f00 / 0x5b00 / 0x0 / 0x0` — те же 4 значения по кругу, затем `-1`.

**Причина A** — Неверный адрес task для SpringBoard:  
`proc_task(sb_proc)` читал `proc_ro + 0x08` как `pr_task`, что давало
неверный адрес. Darksword использует `proc + PROC_STRUCT_SIZE` (0x730) — именно это
совпадает с нашим подтверждённым `our_proc + 0x730 = our_task`.

**Причина B** — Бесконечный цикл без защиты:  
`retryCount` увеличивался только при `validThreadCount == 1`, но когда все TRO
невалидны, `validThreadCount == 0` → `retryCount` не растёт → цикл вечный.

**Причина C** — `rc_probe_tro_offset()` определена, но НИКОГДА не вызывалась.

**Исправления применены**:
- `proc_task()` переписана на `proc + PROC_STRUCT_SIZE` (как `taskbyproc`)
- `maxIter = 1024` добавлен в условие `while`
- `rc_probe_tro_offset(dummyThreadAddr)` вызывается перед обходом
- Диагностический лог PRE-WALK (task addr, sentinel, first_chain, first_tro)
- Проверка `nextChain`: если не kernel ptr и не sentinel — break

#### 2. `vfs_bypass_mac_label` — ИСПРАВЛЕН
**Симптом**: Возвращает -1 для `/var/tmp/TweaksLoader.dylib`.  
**Причина**: `resolvepath()` не находит vnode для файла, только что записанного —
ncache ещё не прогрет.  
**Исправление**: `stat()` warm-up перед вызовом `resolvepath()` + retry через `usleep(5000)` + поиск альтернативного label offset (±16 байт).

#### 3. `/var/jb` — ТРЕБУЕТ ИССЛЕДОВАНИЯ
**Симптом**: `errno = 13` после sandbox escape.  
**Гипотеза**: Sandbox escape неполный, либо `/var` смонтирован read-only до
`remount`. Нужно проверить порядок операций в `kexploit/vfs.m::vfs_mkdir_p`.

В данном документе зафиксированы результаты автоматического поиска патчей (Patch Diffing) между уязвимой (iOS 17.3.1) и пропатченной (iOS 17.4) версиями ядра для iPad8,9.

## Задача
Найти изменения, внесенные Apple в расширение ядра (KEXT) `com.apple.driver.AppleDiskImages2` для закрытия уязвимости CVE-2024-23265, без использования тяжеловесных инструментов (таких как Ghidra или IDA Pro).

## Методология
1. **Автоматическая загрузка:** С помощью скрипта `automate_ipsw.py` были скачаны `kernelcache` для сборок 21D61 (17.3.1) и 21E219 (17.4).
2. **Извлечение KEXT:** Утилита `ipsw` извлекла целевой драйвер `AppleDiskImages2`.
3. **Дизассемблирование:** Был получен полный дамп ассемблерного кода секции `__TEXT_EXEC.__text` для обеих версий.
4. **Python Diffing:** Скрипт `auto_diff.py` разбил код на логические блоки и выявил структурные изменения (разницу в количестве инструкций).

## Результаты: Анализ патча
Особенность бага заключалась в отсутствии проверки указателя на значение `-1` (0xFFFFFFFFFFFFFFFF). В пропатченной версии Apple добавили инструкции `cmn xN, #0x1` (сравнение с -1) и `cbz` (сравнение с 0) перед разыменованием указателей или их использованием.

Скрипт выявил **точные места внедрения патча**.
Пример найденного измененного блока по адресу `0xfffffff008a1904c` (iOS 17.4, 21E219):

```gas
// Загрузка указателя
ldar   x8, [x8]

// Проверка на NULL (добавлено/изменено)
cbz    x8, loc_fffffff008a19108

// Проверка на -1 (специфичный патч CVE-2024-23265)
cmn    x8, #0x1

// Переход, если не равно -1
b.ne   loc_fffffff008a19128

// Безопасное использование
ldp    x11, x10, [x0]
```

## Дальнейшие шаги
Теперь, когда у нас есть полностью работающий механизм дизассемблирования и анализа ядра прямо из Python, мы можем применить этот же подход для **автоматического поиска оффсетов (offsets)** (например, `allproc`, `taskOffset`, `itk_space`), необходимых для джейлбрейка `lara`.

Мы можем искать нужные нам смещения, отталкиваясь от системных вызовов (`syscalls`) и эмулируя выполнение простых инструкций (`ADRP` + `ADD`/`LDR`).
